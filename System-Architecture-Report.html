1. System Overview

High-level understanding of the platform's purpose, stack, and architecture

1.1 Purpose & Domain

LearnHub is a full-featured Online Learning Platform that enables instructors to create and publish courses, students to enroll and learn at their own pace, and administrators to manage the entire ecosystem. The platform supports course content management with lessons, quizzes with auto-grading, progress tracking, enrollment lifecycle management, certificate issuance upon course completion, and instructor review/rating systems.

1.2 Architectural Pattern

The system follows a Microservices Architecture with an API Gateway pattern. Each business domain is isolated into its own independently deployable service with a dedicated database (Database-per-Service pattern). All client traffic flows through a single API Gateway that handles authentication, routing, and CORS. Inter-service communication is synchronous via REST (RestTemplate).

4.2 Key Data Flow: Course Completion Chain

The most complex data flow in the system is the course completion pipeline. When a student passes a quiz, it triggers a chain of synchronous calls across four services:

Student submits quiz → Quiz Service auto-grades
  ↓ if passed
Quiz Service → PUT /api/enrollments/complete → Enrollment Service
  ↓ marks COMPLETED
Enrollment Service → POST /api/certificates/issue → Certificate Service
  ↓ needs student name + course name
Certificate Service → GET /api/students/{id} → Student Service
Certificate Service → GET /api/courses/{id} → Course Service
  ↓ certificate issued with code CERT-2026-0001

4.3 Shared Resources

All 7 microservices share a single MySQL 8.0 instance (run via Docker) but each uses its own isolated schema. There is no direct database-level sharing between services; all cross-service data access happens through REST APIs. The userId field from Auth Service acts as the universal foreign key that links data across Student (studentId), Instructor (instructorId), and all downstream services.

5. Role-Based Access Control (RBAC)

Complete permissions matrix for each role in the system

5.1 Role Definitions

STUDENT

Learners who enroll in courses, track progress, take quizzes, and earn certificates.

Can: Enroll in courses, view own profile/progress, submit quizzes, manage wishlist, review instructors, view own certificates
Cannot: Create/edit/delete courses, manage categories, delete any user, revoke certificates, view other students' data, mark completions

INSTRUCTOR

Course creators who build content, manage lessons, create quizzes, and view their students' performance.

Can: Create/edit/publish/delete own courses and lessons, create quizzes and questions, view course enrollments, view any student's progress/attempts
Cannot: Enroll in courses, submit quizzes, manage wishlists, create categories, delete instructors, revoke certificates, review instructors

ADMIN

System administrators with full platform access for management and moderation.

Can: Everything instructors can do, plus: manage categories, deactivate students, delete instructors, revoke certificates, update any profile, mark completions, view all data
Cannot: Only self-register (must be set via seed data or DB), cannot submit quizzes or review instructors as those are student-only actions

5.3 RBAC Implementation Pattern

Access control is enforced at the controller level using two helper methods present in every controller:

requireRole(userRole, "STUDENT", "ADMIN") → Checks X-User-Role header against allowed roles → Throws 403 FORBIDDEN if role not in allowed list requireOwnershipOrAdmin(userId, resourceOwnerId, role) → If ADMIN, always passes → Otherwise, compares X-User-Id with the resource owner's ID → Throws 403 FORBIDDEN if user does not own the resource

The API Gateway's AuthenticationFilter attaches X-User-Id, X-Username, and X-User-Role headers to every authenticated request after validating the token via the Auth Service's introspect endpoint. These headers are the foundation of the entire RBAC system.
